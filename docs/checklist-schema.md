# Checklist YAML Schema

Every checklist is a single YAML file. The validator in `internal/checklist/validate.go` runs on every file before execution, so all rules below are enforced at load time — including for downloaded third-party checklists.

---

## Top-level fields

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | **yes** | Display name. Max 100 characters. |
| `description` | string | no | Free-text summary shown in listings. |
| `version` | string | no | Arbitrary version string (e.g. `"1.0"`). |
| `type` | string | no | `normal` (default) or `emergency`. |
| `phases` | list | no* | Named groups of items. |
| `items` | list | no* | Flat list of items (no phases). |
| `on_complete` | list | no | Automations to run after the checklist finishes. |
| `trigger_checklist` | string | no | Name of another checklist to start automatically after this one completes. |
| `schedule` | object | no | When this checklist should appear on startup. |

> `phases` and `items` are mutually exclusive — using both is a validation error.
> At least one item (via either field) is required.

### Checklist types

| Value | Behaviour |
|---|---|
| `normal` | Default. Items can be marked N/A; skips are allowed. |
| `emergency` | No N/A, no skips. Every item must be explicitly confirmed. |

---

## Phases

Use `phases` when you want items grouped under named headings.

```yaml
phases:
  - name: CODE        # required, non-empty string
    items:
      - ...           # at least one item per phase
  - name: DEPLOY
    items:
      - ...
```

Rules:
- Every phase must have a non-empty `name`.
- Every phase must contain at least one item.

---

## Items

```yaml
items:
  - id: tests-green           # see ID rules below
    label: All tests passing? # required
    response: Tests confirmed green
    note: Check CI before proceeding
    type: do                  # do | check
    na_allowed: true          # not allowed on emergency checklists
    on_complete:
      - shell: echo "done"
    condition:
      if_yes:
        trigger_checklist: rollback
      if_no:
        skip: true
```

### Item fields

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string | no | Unique identifier. Auto-generated if omitted (`item-0`, `item-1`, …). |
| `label` | string | **yes** | The checklist item text shown to the user. Max 200 characters. |
| `response` | string | no | Pre-filled response hint shown in the input prompt. |
| `note` | string | no | Additional context displayed below the label. |
| `type` | string | no | `do` (default) or `check`. |
| `na_allowed` | bool | no | Whether the user can mark this item N/A. Default `false`. Forbidden on `emergency` checklists. |
| `on_complete` | list | no | Automations that fire when this item is confirmed. |
| `condition` | object | no | Turns this item into a yes/no decision point. |

### Item ID rules

- Letters, digits, hyphens, and underscores only: `[a-zA-Z0-9_-]`
- Maximum 64 characters.
- Must be unique within the checklist (across all phases).
- If omitted, a stable ID is auto-generated by position at load time.

### Item types

| Value | Behaviour |
|---|---|
| `do` | User confirms the step is done. |
| `check` | User must type a value (e.g. a version number, a URL). |

---

## Automations (`on_complete`)

Appears at both item level and checklist level. Each entry fires after the item or checklist completes.

```yaml
on_complete:
  - shell: echo "deployed by $(git config user.name)"
  - webhook: https://hooks.example.com/preflt
  - shell: make notify
    webhook: https://hooks.example.com/preflt  # both on one step is fine
```

### Automation step fields

| Field | Type | Required | Description |
|---|---|---|---|
| `shell` | string | no* | Shell command to run. |
| `webhook` | string | no* | HTTP POST to this URL with a JSON payload. |

> At least one of `shell` or `webhook` is required per step — an empty step is a validation error.

### Webhook rules

- Must start with `http://` or `https://`.
- Receives a POST with a JSON body containing the run state.
- Item-level automations run asynchronously (non-blocking).
- Checklist-level automations run synchronously after the run completes.

---

## Conditions

A condition turns an item into a yes/no branch point.

```yaml
- id: db-migration
  label: Does this release include a database migration?
  condition:
    if_yes:
      trigger_checklist: run-migration
    if_no:
      skip: true
```

### Condition fields

| Field | Type | Required | Description |
|---|---|---|---|
| `if_yes` | branch | no* | Action taken when the user answers yes. |
| `if_no` | branch | no* | Action taken when the user answers no. |

> At least one of `if_yes` or `if_no` is required.

### Branch fields

| Field | Type | Description |
|---|---|---|
| `skip` | bool | Skip this item and continue the checklist. |
| `trigger_checklist` | string | Start another checklist after answering. |
| `abort` | bool | Immediately abandon the current checklist. |

> Every branch must have at least one of `skip`, `trigger_checklist`, or `abort` set. An empty branch is a validation error.

---

## Schedule

Defines when the checklist appears in the startup prompt. This is an advisory hint — the scheduling engine (`schedules.json`) takes precedence for user-managed schedules.

```yaml
schedule:
  frequency: weekly        # daily | weekly | monthly
  on: monday,wednesday     # required for weekly; comma-separated weekday names
  period: morning          # morning | afternoon | evening  (hint only)
  cooldown: 7d             # minimum gap between runs
```

### Schedule fields

| Field | Type | Required | Description |
|---|---|---|---|
| `frequency` | string | **yes** | `daily`, `weekly`, or `monthly`. |
| `on` | string | weekly only | Comma-separated weekday names (e.g. `"monday"`, `"monday,friday"`). |
| `period` | string | no | Time-of-day hint: `morning`, `afternoon`, or `evening`. |
| `cooldown` | string | no | Minimum time between runs. See format below. |

### Cooldown format

| Format | Example | Meaning |
|---|---|---|
| `Nd` | `7d` | N days |
| Go duration | `2h30m` | Standard Go `time.Duration` string |

Valid Go duration units: `ns`, `us`, `ms`, `s`, `m`, `h`.

### Valid weekday names

`monday`, `tuesday`, `wednesday`, `thursday`, `friday`, `saturday`, `sunday`

Case-insensitive. Spaces around commas are ignored.

## What the validator checks

The validator (`Checklist.Validate()`) runs automatically inside `Load()`. All errors are collected before returning, so you see every problem at once.

**Checklist level**
- `name` is non-empty and ≤ 100 characters
- `type`, if set, is `normal` or `emergency`
- `phases[]` and `items[]` are not both non-empty

**Phase level**
- Every phase has a non-empty `name`
- Every phase has at least one item

**Item level**
- At least one item exists in the checklist
- Every item has a non-empty `label` (≤ 200 characters)
- `id`, if set, matches `[a-zA-Z0-9_-]+` and is ≤ 64 characters
- No two items share the same `id` (including across phases)
- `type`, if set, is `do` or `check`
- `na_allowed` is `false` on `emergency` checklists
- Every `on_complete` step has at least one of `shell` or `webhook`
- Every `webhook` value starts with `http://` or `https://`
- `condition`, if set, has at least one branch (`if_yes` or `if_no`)
- Every condition branch has at least one action (`skip`, `trigger_checklist`, or `abort`)

**Schedule level**
- `frequency` is `daily`, `weekly`, or `monthly`
- `on` is provided for `weekly` frequency, with valid weekday names
- `period`, if set, is `morning`, `afternoon`, or `evening`
- `cooldown`, if set, parses as `Nd` or a valid Go duration

## Minimal valid example

```yaml
name: quick-check
items:
  - label: System status green?
  - label: Backups confirmed?
```

Both items get auto-generated IDs (`item-0`, `item-1`) and default `type: do`.

## Full annotated example

```yaml
name: pre-deploy
description: Steps to complete before every production deployment
version: "1.2"
type: normal

on_complete:
  - shell: echo "Deploy checklist completed by $(git config user.name)"
  - webhook: https://hooks.example.com/preflt/deploy

trigger_checklist: post-deploy   # starts automatically after this list finishes

schedule:
  frequency: weekly
  on: tuesday,thursday
  period: morning
  cooldown: 1d

phases:
  - name: CODE
    items:
      - id: tests-green
        label: All tests passing?
        note: Check both unit and integration tests in CI
        type: do

      - id: version-tag
        label: What version are we deploying?
        type: check            # prompts the user to type a value
        response: "Version: "

      - id: migration-check
        label: Does this release include a database migration?
        condition:
          if_yes:
            trigger_checklist: run-migration
          if_no:
            skip: true

  - name: DEPLOY
    items:
      - id: env-vars
        label: Environment variables verified in production?
        type: do

      - id: rollback-plan
        label: Rollback plan documented?
        type: do
        na_allowed: true        # can skip on hotfix deploys
        on_complete:
          - shell: echo "Rollback plan confirmed"
```
